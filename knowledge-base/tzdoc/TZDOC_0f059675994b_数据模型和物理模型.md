# 数据模型和物理模型


## 数据模型


#### 1、什么是数据模型


低代码平台数据模型，是用于定义和管理业务数据结构的核心模块。它像一个“数据蓝图”，决定了平台中数据的存储方式、字段类型以及各数据间的关联关系，是表单、流程等功能能正常运转的基础。


#### 2、数据模型的核心载体——obj_attribute表


在本平台中，数据模型的字段信息主要通过**obj_attribute表**来存储和管理，其核心逻辑如下：


- 字段与表的对应关系：用户在表单设计器中每拖拽添加一个字段（如文本框、下拉框、日期选择器等），系统就会自动在obj_attribute表中生成一行对应的记录。
- 单条记录包含的信息：obj_attribute表的每一行记录，会完整保存该字段的关键属性，例如字段名称、字段类型（文本、数字、日期等）、是否必填、默认值、关联的表单ID等，确保字段的功能和规则可被系统识别。


#### 3、数据模型的作用


- 统一数据结构：通过定义数据模型，让同一业务场景下的数据（如客户信息、订单数据）拥有统一的格式，避免数据混乱。
- 支撑业务功能：表单的展示、数据的录入与查询、流程中的数据传递，都依赖数据模型定义的结构来实现。
- 简化开发操作：用户无需手动编写数据库表结构代码，只需通过可视化操作（拖拽字段）即可完成数据模型的搭建，降低技术门槛。


## 物理模型


#### 1、什么是物理模型


对应业务数据在数据库中实际存在的**物理实体表**，是数据存储的 “最终载体”。它直接映射数据库中的表结构，包含表名、列名、数据类型、主键、外键、索引等数据库层面的具体属性，决定了数据在数据库中如何被实际存储、组织和管理。


简单来说，若将低代码平台的数据体系比作 “房屋”，物理模型就是搭建房屋的 “实体砖瓦”—— 是看得见、摸得着的数据库实际表结构，而非仅停留在设计层面的概念。


#### 2、物理模型的核心特征


-

**与数据库强绑定**：物理模型的结构完全遵循数据库的语法和规则（如 MySQL、Oracle 等），例如数据类型需对应数据库支持的`VARCHAR`、`INT`、`DATE`等，主键、外键约束也需符合数据库的定义规范。

-

**可直接操作数据**：由于是数据库中的实体表，物理模型可通过 SQL 语句直接进行数据的增、删、改、查操作，也能被数据库工具（如 Navicat、DBeaver）直接查看和编辑。

-

**映射业务数据结构**：物理模型会精准映射业务场景中的数据需求，例如 “客户管理” 场景下，会生成`customer`物理表，表中列（如`customer_id`、`customer_name`、`phone`）对应客户的核心信息字段。


## 3、物理模型的作用


-

**保障数据持久化存储**：作为数据的 “最终容器”，物理模型将用户录入的业务数据（如客户信息、订单记录）持久化存储在数据库中，避免数据丢失。

-

**支撑平台功能运行**：低代码平台的表单提交、流程审批、报表统计等功能，最终都需通过操作物理模型（读写数据库表数据）来实现数据的流转和展示。

-

**确保数据一致性**：物理模型通过主键、外键、唯一索引等约束，可强制规范数据格式（如避免重复的客户 ID），保障业务数据的准确性和一致性。


# 数据模型与物理模型关联关系


#### 一、从数据流向维度：“逻辑记录的单向自动同步


数据模型与物理模型的关联，本质是**“逻辑层数据”向“存储层数据”的单向自动流转**，以用户拖拽字段（如FIELD1）为例，完整数据流向链路如下：


- **源头：用户操作触发逻辑数据生成**
用户在表单设计器拖拽字段，系统首先在数据模型的`obj_attribute`表中生成该字段的逻辑记录（含字段名称、类型、必填规则等属性），此时数据仅停留在“逻辑定义层”，未进入数据库存储。
- **流转：平台监听并同步物理数据**
平台实时监测`obj_attribute`表的新增记录，当识别到逻辑数据后，自动将其转化为符合数据库语法的物理数据规则（如文本类型→`VARCHAR`、必填规则→`NOT NULL`约束）。
- **终点：物理宽表落地存储**
转化后的物理规则直接作用于对应的物理宽表，在表中新增同名字段（如FIELD1），完成“逻辑数据”到“物理存储数据”的最终落地，且整个过程无需人工编写SQL或手动操作数据库。


简言之，数据流向是**“用户操作→obj_attribute表（数据模型）→物理宽表（物理模型）”**，呈现“逻辑先行、物理跟随”的单向同步特征。


#### 二、从业务层面维度：“低代码操作→功能支撑”的双向价值关联


从业务视角看，二者的关联是低代码平台实现“易用性”与“功能性”的核心保障，具体体现在两个方向的价值：


-

**对用户操作：降低技术门槛，实现“所见即所得”**


  - 用户无需掌握数据库知识，仅通过可视化拖拽（设计数据模型逻辑），即可自动完成物理表字段创建，避免手动建表、写约束的复杂操作；
  - 例如用户定义“订单状态”字段（数据模型）后，无需关心`VARCHAR`类型配置、非空约束设置，平台会自动在物理表中落地对应字段，让业务设计与存储落地无缝衔接。


-

**对平台功能：支撑业务数据全链路运转**


  - 数据模型的逻辑定义为表单展示、数据校验提供规则（如必填字段在表单中提示用户），物理模型的存储能力为数据持久化、流程流转提供载体（如表单提交后数据存入物理表，流程审批时读取物理表数据）；
  - 二者协同确保业务功能闭环：例如“订单提交”功能中，数据模型校验“订单金额”为必填（逻辑层），物理模型存储“订单金额”数据（存储层），共同支撑订单数据的录入、保存与后续查询统计。
