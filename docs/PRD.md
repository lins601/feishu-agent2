# 制造知识助手 · PRD

> 版本：v4.4 | 日期：2026-06-09

---

## 一、产品概述

### 1.1 一句话

公司有三套制造系统（WMS/QMS/MES），产线每天产生大量异常咨询。老员工在飞书群被反复 @ 回答同样问题。**制造知识助手 = 把 SOP 文档存入 Wise 知识库，飞书里 @ 机器人用自然语言提问，3 分钟内拿到带步骤和图片的结构化答案。**

### 1.2 类比理解

```
4S 店维修车间       →   电子制造工厂
发动机/底盘/电气     →   WMS/QMS/MES
故障码 P0171        →   "IQC 检验单提交失败"
维修手册             →   MinDoc Wiki 文档
老师傅翻手册 10 分钟  →   AI 助手 3 分钟返回标准步骤
```

### 1.3 目标用户

| 角色 | 痛点 | 期望 |
|------|------|------|
| 产线操作员 | 遇到报错不会处理，等老师傅回复 20 分钟 | @机器人 3 分钟拿到操作步骤 |
| 新入职员工 | 不懂业务流程和状态机流转 | 问一句知道"卡在哪、下一步谁审批" |
| 老员工/导师 | 被反复 @ 回答同样问题，打断工作 | 高频问题收录后不再被打扰 |
| IT/开发 | 排查时缺业务上下文 | 拿到 devMessage 排查路径，定位更快 |

### 1.4 权衡目标

| 目标 | 优先级 | 说明 |
|------|--------|------|
| 准确率 | **P0** | 答错了产线可能停工，宁可不答不能瞎答 |
| 覆盖度 | P1 | WMS/QMS/MES 高频问题都得收录 |
| 知识保鲜 | P1 | 系统升级后旧答案必须及时标记或下线 |
| 响应速度 | P1 | 飞书消息 3 分钟内必须有回复 |

## 二、业务流转详解

### 2.1 WMS · 仓储管理系统

**管**：物料从供应商到仓库再到产线，全程追踪库存和位置。

**核心流程**：

```
① 入库：收货通知 → 到货扫描 → 质检判定 → 上架确认 → 库存更新（入库单/上架单）
② 出库：领料申请 → 波次创建 → 拣货任务 → 拣货确认 → 发料出库（出库单/拣货单）
③ 库内：库存查询 → 移库移位 → 盘点差异 → 库存调整（移库单/盘点单）
```

**状态机**：`入库单：待收货→收货中→已收货→部分上架→已上架→已关闭`

**常见异常**：入库单提交不了、库存对不上、扫码扫不出批次

### 2.2 QMS · 品质管理系统

**管**：从来料检查到生产过程到出货，全链路品质把关。

**核心流程**：

```
① IQC来料：来料登记→抽样(AQL)→检验→判定→合格入库/退货/让步（IQC检验单）
② IPQC制程：首件确认→巡检→缺陷记录→触发异常单（巡检记录/异常单）
③ OQC出货：出货申请→抽样→功能测试→判定放行→打印COA（OQC检验单）
④ 异常处理：问题发现→围堵→根因分析→纠正措施→预防措施→关闭（8D报告/CAPA）
```

**状态机**：`IQC检验单：待检验→检验中→已判定→合格入库/不合格退货/让步接收→已关闭`

**常见异常**：IQC 检验单提交报错（TOP1 高频）、不合格不知退货流程、8D 超期未关闭

### 2.3 MES · 制造执行系统

**管**：从工单下发到产线生产到完工入库，跟踪生产进度。

**核心流程**：

```
① 工单管理：计划下达→工单创建→BOM展开→物料齐套→工单发放（生产工单）
② 排产排程：产线评估→排产计算→看板发布→产线调配→插单处理（排产计划）
③ 生产执行：上料确认→首件检查→过站生产→质量检测→报工（报工记录）
④ 完工入库：完工确认→OQC检验→包装入库→工单关闭
```

**状态机**：`工单：待投产→已排产→生产中→PAUSE→生产中→已完工→已入库→已关闭`

**常见异常**：工单发不出去、报工提交不了、SMT 贴错料

### 2.4 三系统协作

```
供应商 → WMS入库 → QMS IQC检验
  → WMS发料 → MES领料生产 → QMS IPQC巡检
  → QMS OQC检验 → WMS成品入库 / QMS 8D异常处理
```

**跨系统知识要点**：
- IQC 不合格退货：QMS 判定 + WMS 执行，知识需跨系统提醒
- 报工发现不良：MES 记录 + QMS 异常单，知识需跨系统操作步骤
- 批次追溯：WMS 批次 + QMS 检验记录，知识需说明两系统数据对应关系

## 三、功能需求

### 3.1 用户故事

```
US-01  产线操作员在飞书群里 @制造小助手提问
       → 3 分钟内收到结构化回复（原因 + 步骤 + 关联系统 + 来源）

US-02  操作员用不同说法问同一个问题
       → "IQC 单子交不上去" 和 "IQC 检验单提交失败" 命中同一条知识

US-03  检索未命中时
       → 回复"暂未收录，已记录您的问题"，问题自动进入待补充队列

US-04  多条结果置信度接近时
       → 列出 Top-3 让用户选择

US-05  操作员对答案回复"有用"或"没用"
       → 反馈写入对话记录，低分知识条目标记"待审核"

US-06  知识管理员在后台查看/编辑/标记过期知识条目
       → 最简管理后台，能 CRUD

US-07  新知识自动沉淀
       → 飞书群聊里被回复的问题 → LLM 提取为 Skill Schema → 人工确认后入库（Wise 知识库 / 自建库）

US-08  过期知识自动标记
       → 系统升级后，关联旧版本的知识条目标记"待审核"
```

### 3.2 核心流程

```
用户提问
  │
  ├── 飞书 WebSocket 长连接接收
  ├── 短期记忆拼接上下文（最近 3 轮完整对话 + 历史摘要压缩）
  ├── 意图识别（规则关键词匹配，识别 WMS/QMS/MES + 提取关键词）
  │
  ├── 缓存命中（Caffeine 热点缓存）→ 直接返回
  │
  ├── 未命中 → 混合检索：
  │     ├── 关键词检索（tsvector + GIN 索引，中文分词）
  │     ├── 语义检索（pgvector 余弦相似度，1536 维向量）
  │     └── 合并去重 + 加权排序（α=0.6）→ Top-3
  │
  ├── 回答组装（RichTextBuilder）：
  │     ├── 按置信度分 4 档（≥0.8 / 0.6~0.8 / <0.6 / 无结果）
  │     ├── 将 source_text 转为飞书富文本格式：文字段 + 内联图片
  │     ├── 图片直接通过 image_key 引用，无需手动拆段拼接
  │     └── 低置信度时添加"⚠️ 仅供参考"前缀
  │
  └── 回复飞书富文本消息（post 类型，原生支持内联图片）
      → 异步：写缓存 + 记对话 + 更新会话摘要 + 更新用户画像 + 记录未命中问题

[任何环节异常 → 降级兜底（详见 §9）→ 4 级兜底话术（§9.10）→ 用户始终有回复]
```

**关键技术决策**：

| 决策 | 选型 | 理由 |
|------|------|------|
| 意图识别 | 规则关键词（~1ms） | 确定性高、速度快、WMS/QMS/MES 关键词几乎不重叠 |
| 答案组装 | 富文本拼接（~3ms） | 飞书 post 消息原生支持内联图片，无需复杂 JSON 卡片拼接 |
| 消息处理 | @Async 异步 | 飞书要求 3 分钟回复，异步先确认再执行，避免超时 |
| 检索降级 | 单路×0.7 + 强制提示 + 多级兜底 | 防止低质量结果冒充高置信度答案（详见 §9.4） |

### 4.2 知识入库管道

**主方案：Wise 知识库 → 本地库同步**

```
Wise 知识库 → Wise API 定时拉取
  │
  ① 文档获取：通过 Wise API 获取指定知识库空间下的文档列表，检测新增/更新
  ② 文字处理：HTML → 纯文本段落，图片位置标记为 [img:ref|alt]
  ③ 图片处理：提取 <img src> → 下载原始图片 → 上传飞书 CDN → 获取 image_key
  ④ 生成 Embedding：调 OpenAI API → 1536 维向量（LRU 缓存减少重复调用）
  ⑤ 写入 PG：
     ├── knowledge_source（含 trigger_keywords, source_text, embedding, image_refs）
     ├── image_mapping（wise_doc_id → image_key 映射）
     └── source_sync_log（同步状态）
  ⑥ 异常处理：
     ├── 单篇文档失败 → 跳过，标记 partial
     ├── 图片失败 → 跳过该图，不阻塞入库
     ├── 内容未变（raw_content_hash 一致）→ 跳过
     └── 整次同步失败 → 保留旧数据，飞书告警
```

**备选方案：MinDoc 爬取 → 本地处理**（详见 §七 知识库策略切换条件）

```
MinDoc AJAX API → Jsoup 解析 → 同上步骤 ②-⑥
```

### 4.3 技术栈

| 层 | 技术 | 说明 |
|----|------|------|
| 消息接收 | 飞书 WebSocket 长连接 | 仅此模式，无需公网 URL，SDK 自带重连 |
| 应用框架 | Spring Boot 3.4 + JDK 17 | 全部业务逻辑 |
| 数据存储 | PostgreSQL 16 + pgvector | 知识源 + 运营数据 + 向量索引 |
| 关键词检索 | tsvector + GIN 索引 | 中文分词（zhparser/pg_jieba），中文词语级匹配 |
| 语义检索 | pgvector 余弦相似度 | ivfflat 索引（lists=100），1536 维 |
| Embedding | OpenAI text-embedding-ada-002 | LRU 缓存 500 条，减少重复调用 |
| 意图识别 | IntentClassifier 规则匹配 | WMS 26 词 / QMS 20 词 / MES 16 词，~1ms |
| 答案组装 | RichTextBuilder 富文本拼接 | 飞书 post 消息原生支持内联图片，~3ms |
| 热点缓存 | Caffeine | 1000 条，30 分钟 TTL，SHA-256 Key |
| 短期记忆 | SessionContextService | 最近 3 轮完整 + 旧对话摘要压缩，30min 超时 |
| 用户画像 | user_profile 表 | 偏好系统 + 高频问题，每 10 轮异步更新 |
| 爬取 | Wise 知识库 API（主）/ MinDoc AJAX API（备） | Token 鉴权，Jsoup HTML 解析 |
| 图片上传 | FeishuImageService | 下载 → 飞书 CDN → image_key → image_mapping 表 |
| 小C导入 | Apache POI / OpenCSV | 解析 CSV/Excel 表格，无 API |

## 五、知识库策略

> **核心思路**：优先利用公司已有的 Wise 知识库平台，以文档形式存放知识内容，降低自建成本。自建知识库作为备选方案，当 Wise 无法满足需求时启用。

### 5.1 主方案：Wise 知识库（已有平台）

将 SOP 文档、操作指南、异常处理手册等知识内容，以结构化文档形式存入公司现有的 Wise 知识库（docs.cvte.com）。

| 维度 | 说明 |
|------|------|
| 知识存放 | SOP 文档直接上传到 Wise 知识库，按 WMS/QMS/MES 分类管理 |
| 内容维护 | 业务专家直接在 Wise 上编辑/新增/更新文档，无需开发介入 |
| 系统对接 | 通过 Wise API 定时拉取文档 → 本地 PG 建索引 → 机器人检索本地索引返回答案 |
| 图片管理 | 图片嵌入 Wise 文档，系统拉取时自动下载并上传飞书 CDN |
| 更新同步 | Wise 文档更新后，下次定时同步自动拉取最新版本 |

**优势**：

- 无需自建知识库平台，复用公司已有基础设施
- 业务专家可自主维护内容，降低对开发的依赖
- Wise 自带版本管理、权限控制、审批流程
- 与现有 MinDoc 文档体系无缝衔接（Wise 即 MinDoc 企业版）

### 5.2 备选方案：自建知识库

当 Wise 知识库无法满足检索精度或功能需求时，启用完整的自建方案（即本 PRD 所描述的架构）：

| 维度 | 说明 |
|------|------|
| 存储 | PostgreSQL 16 + pgvector，知识条目 + 1536 维向量同表存储 |
| 入库管道 | Wise/MinDoc 爬取 → HTML 解析 → Embedding 生成 → 写入本地库 |
| 检索 | 混合检索（关键词 tsvector + 语义 pgvector，α=0.6） |
| 管理后台 | 知识 CRUD + 状态管理 + 质量审核 |

**优势**：

- 完全掌控检索逻辑，支持置信度分档、降级兜底等高级策略
- 向量检索 + 关键词检索混合，召回率和准确率更优
- 可定制触发关键词、跨系统关联、图片穿插等结构化字段

### 5.3 切换条件

明确以下场景触发从主方案切换到备选方案（或混合使用）：

| # | 切换条件 | 说明 | 动作 |
|---|---------|------|------|
| C1 | Wise API 检索能力不足 | Wise 自带搜索无法满足关键词+语义混合检索需求 | 启用本地 PG 索引层，Wise 仅作为数据源 |
| C2 | 语义检索需求 | 需要向量检索覆盖用户换说法场景，Wise 不支持 embedding | 本地生成 pgvector 向量，Wise → 本地库同步 |
| C3 | Wise API 可用性低 | API 频繁超时/限流，影响 3 分钟回复 SLA | 切换到本地库全量缓存，定时增量同步 |
| C4 | 检索性能不达标 | Wise 搜索响应 > 1s，加上组装时间超出 SLA | 本地 PG 索引（关键词 ~5ms + 向量 ~50ms） |
| C5 | 需要高级运营能力 | 需要置信度分档、反馈闭环、差评聚类等运营功能 | 本地库存储反馈数据，Wise 仍为内容源 |

**推荐推进路径**：

```
Phase 1：Wise 知识库 + 本地 PG 索引（混合模式）
  → Wise 作为知识内容管理源（人工维护）
  → 定时同步到本地 PG（自动生成关键词 + embedding）
  → 机器人检索本地 PG 索引（保证性能和精度）
  → 兼顾「业务专家自主维护」和「高质量检索」

Phase 2：评估是否需要纯自建
  → Phase 1 运行 1-2 个月后，根据实际运营数据决定
  → 若 Wise 管理效率足够 → 维持 Phase 1 架构
  → 若需要更细粒度控制 → 逐步迁移到纯自建方案
```

> **结论**：当前推荐 Phase 1 混合模式——Wise 管内容，本地 PG 管检索。既利用已有平台降低维护成本，又保留自建检索引擎保证回答质量。纯自建方案作为 Phase 2 备选。

## 六、主动反馈与定期人工巡检 SOP

> 知识库上线后，需要通过**用户主动反馈**和**定期人工巡检**形成闭环，持续保障知识质量和覆盖度。

### 6.1 主动反馈机制（自动触发）

| 触发方式 | 触发条件 | 系统动作 |
|---------|---------|---------|
| 用户点踩 | 操作员对回复点「没用」 | 记录反馈到 conversation_log，knowledge_id 关联差评计数 |
| 连续差评 | 同一知识条目连续 3 次「没用」 | 自动标记 status='pending_review' + 飞书通知知识管理员 |
| 未命中问题 | 检索无结果 | 自动写入 unanswered_question 表 + 每周汇总通知管理员 |
| 高频重复问 | 同一问题 24h 内被问 ≥ 5 次 | 标记为热点问题 + 检查现有知识是否覆盖充分 |
| 用户追问 | 用户对回复继续追问 | 记录为「答案不够清晰」信号，关联知识条目标记关注 |
| 低置信度命中 | 综合分 0.6~0.8 区间被命中 | 记录为「答案可能不够准确」，累计 5 次触发审核 |

### 6.2 定期人工巡检 SOP

#### 6.2.1 周度巡检（每周一，约 30 分钟）

| # | 检查项 | 操作 | 产出 |
|---|--------|------|------|
| W1 | 差评条目审核 | 查看所有 pending_review 状态的知识条目 | 确认差评原因：答案错误 → 修正；表述不清 → 优化；流程变更 → 更新 |
| W2 | 未命中问题 Top-10 | 查看本周 unanswered_question 按频次排序 | 高频未命中 → 创建新知识条目（Wise 上编辑 → 触发同步） |
| W3 | 热点问题复查 | 检查 24h 内被问 ≥ 5 次的问题对应知识 | 确认答案是否仍然准确，是否需要补充说明 |

#### 6.2.2 月度巡检（每月第一个工作日，约 2 小时）

| # | 检查项 | 操作 | 产出 |
|---|--------|------|------|
| M1 | 知识覆盖度分析 | 统计各系统（WMS/QMS/MES）知识条目数 vs 未命中数 | 识别知识盲区，制定本月补充计划 |
| M2 | 过期知识排查 | 检查 last_synced_at > 90 天未更新的知识条目 | 联系业务专家确认是否仍有效，过期则标记 expired |
| M3 | 触发关键词优化 | 分析未命中问题的用词习惯 | 补充 trigger_keywords 变体，提高关键词召回率 |
| M4 | 跨系统关联检查 | 检查 cross_system 字段是否完整 | 补充 WMS↔QMS、MES↔QMS 等跨系统操作提醒 |
| M5 | 反馈闭环率 | 统计：差评条目中已修正/已确认有效的比例 | 目标 ≥ 90%，低于则加大力度 |

#### 6.2.3 季度巡检（每季度首月，约半天）

| # | 检查项 | 操作 | 产出 |
|---|--------|------|------|
| Q1 | 全量知识审计 | 遍历所有 active 状态条目，逐条确认有效性 | 过期 → expired；错误 → 修正；确认有效 → 更新 reviewed_by |
| Q2 | 系统升级影响评估 | 与 IT 确认 WMS/QMS/MES 近期升级计划 | 预判受影响的知识条目，提前标记 pending_review |
| Q3 | 用户满意度趋势 | 统计月度「有用」率变化曲线 | 识别下降趋势，定位原因（知识老化？新问题增多？） |
| Q4 | 知识库架构评估 | 评估当前 Wise + 本地 PG 混合模式是否仍适用 | 若满足需求 → 维持；若不足 → 启动纯自建方案评估 |

## 八、混合检索设计

### 8.1 为什么必须混合检索

| 场景 | 纯关键词 | 纯语义 | 混合 |
|------|---------|--------|------|
| "IQC 检验单提交失败"（标准问法） | ✅ 命中 | ✅ 命中 | ✅ |
| "IQC 单子交不上去"（换说法） | ❌ "交不上去"不匹配 | ✅ 语义理解≈提交失败 | ✅ 语义路兜住 |
| "缓存击穿"（专业术语≠一般理解） | ✅ 精准命中专业词汇 | ❌ 可能理解成"Redis缓存空值" | ✅ 关键词纠正 |
| Embedding API 挂了 | ✅ 本地可检索 | ❌ 完全不可用 | ✅ 关键词降级 |

**结论**：关键词保专业术语准确率，语义保自然语言变体覆盖，两条路互相兜底。

### 8.2 权重 α=0.6 的理由

| 理由 | 说明 |
|------|------|
| 专业术语多 | "IQC""OQC""AQL""NCR" 语义空间距离远，关键词精确匹配 |
| 同词不同义 | "状态不允许"在 WMS 和 QMS 含义不同，关键词+系统标签才能区分 |
| 问法有变体 | "提交不了""交不上去""报错了" ≈ 同义，语义覆盖比关键词好 |

### 8.3 召回率与幻觉解决方案

**四重保障**：

```
保障 1：关键词本身优化
  ├── 中文分词（zhparser）："IQC单子交不上去" → ["IQC","单","交"]
  └── trigger_keywords 存多种问法变体 → 录入时覆盖

保障 2：语义检索兜底
  └── 关键词未命中 → pgvector 同义匹配 → "交不上去"≈"提交失败"

保障 3：置信度分级防幻觉（见 §10.2）

保障 4：降级时防幻觉（见 §9）
```

## 九、降级与兜底

> **核心原则**：用户发消息后必须收到回复，任何环节出问题都不能让用户"石沉大海"。降级顺序：精准回答 → 模糊回答 → 候选列表 → 兜底话术 → 人工转接。

### 9.1 消息接收层兜底

| 场景 | 症状 | 策略 |
|------|------|------|
| WebSocket 断连 | 消息收不到 | SDK 自带重连 + 心跳检测（30s 间隔）+ 断连告警飞书群通知 IT |
| WebSocket 重连失败 | 持续断连 | 重试 5 次指数退避（1s→2s→4s→8s→16s），仍失败则切换备用连接或飞书轮询模式 |
| 消息重复投递 | 同一消息收到多次 | 基于 message_id 幂等去重，5 分钟内相同 message_id 直接丢弃 |
| 消息格式异常 | JSON 解析失败 | try-catch 捕获 → 回复"消息解析异常，请重新发送" → 记录原始消息到异常日志 |
| 飞书 API Token 过期 | API 返回 401/403 | 自动刷新 token，失败则用缓存的上一个有效 token 发兜底消息，同时告警 |
| 飞书 API 限流 | 429 Too Many Requests | 指数退避重试（1s→4s→9s），3 次仍失败则排队延迟发送，优先保障 P0 用户 |
| 飞书消息发送超时 | 回复发不出去 | 3 分钟超时 → 先发"正在查询中…"占位 → 异步重试（最多 3 次） → 最终失败则记录待补发 |
| 飞书消息体过大 | 超过富文本 30KB 限制 | 自动截断 + 末尾添加"内容过长已截断，请追问获取更多细节" |
| 收到非文本消息 | 用户发图片/语音/文件 | 回复"暂仅支持文字提问，请用文字描述您的问题 🙏" |
| 收到空消息或纯表情 | 无有效文本内容 | 忽略不回复，不记录为未命中 |
| 消息积压（突发流量） | 队列堆积 > 100 条 | 启用排队机制：先回复"当前咨询较多，预计 X 分钟后回复您"，按 FIFO 处理 |

### 9.2 理解层兜底

| 场景 | 症状 | 策略 |
|------|------|------|
| 意图识别无匹配 | 关键词全部未命中 | 不强制归类，直接走全库语义检索（system 字段不参与过滤） |
| 意图误判 | 用户问 WMS 被识别为 QMS | 检索结果置信度 < 0.6 时，自动放宽 system 过滤条件重搜一次 |
| 意图识别超时 | 规则匹配异常 | 设置 50ms 超时，超时则跳过意图识别，走全库检索 |
| 短期记忆读取失败 | SessionContextService 异常 | 跳过上下文拼接，仅用当前问题检索，回复不带上下文关联 |
| 短期记忆超 token | 3 轮对话 + 摘要超过 4096 token | 递归压缩：先压摘要（保留关键实体），再截断最早一轮对话，最后仅保留当前问题 |
| 会话超时（30min） | 用户间隔过长后再次提问 | 清空上下文，视为新会话，回复"已开启新对话，请描述您的问题" |
| 摘要压缩失败 | LLM 压缩调用异常 | 直接丢弃旧对话，仅保留最近 1 轮 + 当前问题 |
| 用户画像读取失败 | user_profile 查询异常 | 跳过画像加分，按默认权重检索，不影响主流程 |

### 9.3 缓存层兜底

| 场景 | 症状 | 策略 |
|------|------|------|
| 缓存未命中 | 正常 miss | 走检索流程，结果写入缓存 |
| 缓存穿透 | 用户反复问不存在的问题 | 对"无结果"也缓存（空结果标记，TTL=5min），防止反复打到 DB |
| 缓存击穿 | 热点 key 过期瞬间大量并发 | Caffeine `refreshAfterWrite` 异步刷新，旧值继续服务直到新值就绪 |
| 缓存雪崩 | 大量 key 同时过期 | TTL 加随机抖动（30min ± 5min），避免集中过期 |
| 缓存组件异常 | Caffeine 内部错误 | try-catch 跳过缓存，直接走检索流程（降级不降功能） |
| 缓存结果已过期知识 | 知识更新后缓存未刷新 | 知识入库时主动清除相关缓存 key（按 source_id 匹配） |
| 缓存容量满 | 超过 1000 条 | Caffeine LRU 自动淘汰最久未访问的条目，无需人工干预 |

### 9.4 检索层兜底

| 场景 | 症状 | 策略 |
|------|------|------|
| 双路正常 | — | 关键词 + 语义合并，α=0.6 |
| 语义路不可用 | Embedding API 超时/报错 | 仅关键词检索，置信度×0.7，消息顶部标注"⚠️ 降级模式（语义暂不可用）" |
| 关键词路不可用 | PG tsvector 查询异常 | 仅语义检索，置信度×0.7，消息顶部标注"⚠️ 降级模式（关键词暂不可用）" |
| 双路都不可用 | PG 连接失败 | 查缓存中是否有相似问题的历史结果 → 有则返回并标注"历史结果" → 无则回复"系统繁忙，请稍后重试，已通知技术人员" + 飞书告警 |
| Embedding API 限流 | 429 或超时 | 重试 2 次（1s→4s），仍失败则走纯关键词降级 |
| Embedding API 返回异常向量 | 向量维度不对或全零 | 校验向量维度和范数，异常则丢弃该向量，走纯关键词降级 |
| 检索结果为空 | 两路都无结果 | 回复"暂未收录该问题，已记录您的问题并提交专家审核" + 写入 unanswered_question + @管理员 |
| 检索结果全为低分 | 最高分 < 0.6 | 展示 Top-3 候选列表，让用户自行选择，标注"以下为可能相关的内容，请确认" |
| 数据库查询超时 | 查询超过 2 秒 | 设置 2s statement_timeout，超时则走缓存兜底 + 告警 |
| 检索结果包含过期知识 | status=expired 的记录被检出 | 过滤掉 expired 状态，仅返回 active + pending_review，pending_review 结果标注"⚠️ 该知识待审核" |
| pgvector 索引损坏 | ivfflat 查询报错 | 回退到暴力搜索（sequential scan），性能降级但功能正常 + 告警 |

### 9.5 生成层兜底

| 场景 | 症状 | 策略 |
|------|------|------|
| 富文本构建失败 | RichTextBuilder 异常 | 降级为纯文本回复：直接输出 source_text（去掉图片引用） |
| 纯文本也失败 | 字符串处理全部异常 | 返回固定兜底话术"系统暂时无法生成回答，请稍后重试或联系管理员（错误码：{uuid}）" |
| image_mapping 查询失败 | 图片映射表不可用 | 跳过所有图片，仅输出文字段落，不影响文字内容 |
| image_key 为空 | 飞书未返回图片 key | 跳过该图片，文字流不中断 |
| 图片上传飞书 CDN 失败 | 上传 API 报错 | 跳过该图片 + 文字占位 `[图片暂不可用]`，不阻塞消息发送 |
| 富文本超过飞书大小限制 | 单条消息 > 30KB | 自动截断步骤详情，保留前 3 步 + "更多步骤请追问" |
| 置信度分档边界模糊 | 0.79 vs 0.81 差异小 | 增加缓冲带：0.78~0.82 区间统一添加"仅供参考"标注 |

### 9.6 异步处理层兜底

| 场景 | 症状 | 策略 |
|------|------|------|
| @Async 线程池满 | 队列堆积 | 队列满时同步执行（牺牲响应速度保功能），飞书告警扩容 |
| 对话记录写入失败 | DB 写入异常 | 重试 2 次，仍失败则写入本地文件日志（定时回扫），不影响用户回复 |
| 会话摘要更新失败 | 异步摘要压缩失败 | 下次请求时惰性重建，用户无感知 |
| 用户画像更新失败 | 异步更新异常 | 跳过本次更新，等待下一个 10 轮周期 |
| 未命中问题记录失败 | unanswered_question 写入异常 | 写入本地文件 + 告警，定时任务回扫入库 |
| 缓存写入失败 | Caffeine put 异常 | 跳过缓存写入，下次仍走检索，不影响本次回复 |

### 9.7 爬虫入库层兜底

| 场景 | 症状 | 策略 |
|---------|------|------|
| MinDoc 登录失败 | Token 失效 / 401 | 自动重新登录获取新 Token，3 次失败则告警 + 暂停爬取 |
| MinDoc API 返回空 | 接口无数据 | 区分"空数据"和"网络错误"：空数据告警人工检查，网络错误走重试 |
| 单篇文档解析失败 | HTML 结构变化 | 跳过该文档，标记 partial，继续爬下一篇 + 记录失败原因 |
| 整次爬取失败 | 网络全面不通 | 保留旧数据不动，飞书告警，不自动重试（防止脏数据覆盖好数据） |
| 图片下载失败 | 404 / 超时 | 跳过该图，重试 2 次，仍失败则记录 image_key 为空，不阻塞内容入库 |
| 图片上传飞书 CDN 失败 | 飞书 API 报错 | 跳过该图 + 文字占位，记录 source_url 以便后续补传 |
| 网络超时 | 连接超时 | 3 次指数退避（1s→4s→9s），仍超时则跳过当前文档 |
| 内容哈希一致 | raw_content_hash 未变 | 跳过该文档（内容没变），更新 last_synced_at 时间戳 |
| Embedding 生成失败 | OpenAI API 报错 | 该文档 embedding 置空，仅支持关键词检索，标记"待补向量" + 后续定时任务补生成 |
| 数据库写入失败 | PG 异常 | 单篇重试 3 次，仍失败则跳过 + 告警；整批失败则回滚事务 + 保留旧数据 |
| 爬取到重复内容 | source_id 冲突 | ON CONFLICT DO UPDATE 覆盖更新，记录更新日志 |
| 爬取内容含敏感信息 | 检测到密码/Token 等 | 正则过滤敏感字段，替换为 `[已过滤]`，告警通知人工确认 |
| 爬取结构大幅变化 | 连续 > 50% 文档失败 | 自动暂停爬取 + 告警"MinDoc 结构可能变化，请人工检查" |

### 9.8 飞书交互层兜底

| 场景 | 症状 | 策略 |
|------|------|------|
| 用户快速连续提问 | 同一用户 2 秒内发多条 | 合并处理：取最后一条作为有效问题，前面回复"收到，正在处理您最新的问题" |
| 用户发送超长文本 | 问题 > 2000 字 | 截断前 500 字 + "…" 作为检索输入，回复中提示"您的问题较长，已基于前部分内容回答" |
| 用户消息含特殊字符 | SQL 注入 / XSS 尝试 | 输入清洗：去除 SQL 特殊字符，转义 HTML 标签，防止注入 |
| 用户 @ 了多个机器人 | 同时 @ 多个 bot | 检测到其他 bot 也被 @ 时正常回复，不做特殊处理 |
| 群聊 vs 私聊 | 不同场景权限不同 | 私聊直接回复；群聊需要 @ 机器人才触发，避免刷屏 |
| 用户对机器人回复再回复 | 回复富文本消息而非新提问 | 识别为追问，拼接到上下文后重新检索 |
| 用户反馈"有用/没用" | 通过追问文本触发反馈 | 识别"有用""没用""👍""👎"等关键词，写入 feedback 字段 |
| 反馈数据写入失败 | DB 异常 | 记录到本地日志，定时回扫，回复不受影响 |

### 9.9 基础设施层兜底

| 场景 | 症状 | 策略 |
|------|------|------|
| PostgreSQL 连接池耗尽 | 获取连接超时 | 连接池上限 20，超时时走缓存兜底 + 告警，非核心查询（日志/画像）延迟执行 |
| PostgreSQL 主库故障 | DB 完全不可用 | 切换只读副本（如有），仅支持查询；无副本时回复"系统维护中" + 最高级别告警 |
| 应用 OOM | 内存溢出 | JVM 参数 -XX:+HeapDumpOnOutOfMemoryError + K8s 自动重启 + 告警 |
| 磁盘满 | 日志写不进去 | 日志自动轮转（按天 + 按大小），保留最近 7 天 + 告警 |
| 应用重启/发布 | 服务短暂不可用 | 飞书 WebSocket 自动重连 + 未处理消息重新拉取 + 用户侧提示"服务已恢复" |
| 定时任务冲突 | 多个定时任务同时执行 | 加分布式锁（DB advisory lock），确保同一时间只有一个任务执行 |

### 9.10 全局兜底话术

> 当所有降级策略都失败时的最终兜底，保证用户**永远有回复**。

| 兜底级别 | 触发条件 | 回复话术 |
|---------|---------|---------|
| L1 · 降级回复 | 检索降级但仍能返回结果 | "⚠️ 降级模式：[回答内容]（当前系统部分功能受限，结果可能不够完整）" |
| L2 · 无结果 | 检索无匹配 | "暂未收录该问题，已记录您的问题并提交专家审核 📝 您也可以尝试换一种方式描述" |
| L3 · 系统繁忙 | 检索 + 缓存都不可用 | "系统繁忙，请稍后重试 ⏳ 已通知技术人员处理，预计 X 分钟恢复" |
| L4 · 最终兜底 | 所有组件都不可用 | "抱歉，系统暂时无法服务，请联系管理员或拨打 IT 热线：{phone} 📞（错误码：{uuid}）" |

### 9.11 告警与监控

| 告警项 | 触发条件 | 告警方式 |
|--------|---------|---------|
| 消息积压 | 待处理队列 > 50 条 | 飞书群通知 IT |
| 检索降级 | 连续 5 分钟走降级模式 | 飞书群通知 |
| 爬虫失败率高 | 单批次失败 > 50% | 飞书群通知 + 自动暂停 |
| Embedding API 异常 | 连续 3 次调用失败 | 飞书群通知 |
| DB 连接池紧张 | 活跃连接 > 80% | 飞书群通知 |
| 未命中问题激增 | 1 小时内未命中 > 20 条 | 飞书群通知管理员补充知识 |
| 用户差评集中 | 同一知识条目连续 3 次"没用" | 标记"待审核" + 通知知识管理员 |
| 服务完全不可用 | L4 兜底触发 | 飞书群 + 短信通知 IT |

## 十、回答规范

### 10.1 回答内容结构（飞书富文本消息）

```
📋 IQC 检验单提交时提示"数据不存在或已删除"

原因：
  1. 数据被其他人删除了
  2. 页面没有刷新到最新状态

处理步骤：
  1. 刷新页面，重新搜索这条数据
  2. 确认是否由同事刚刚删除
  [📷 刷新按钮位置示意图]     ← 飞书富文本原生内联图片
  3. 如果数据消失，联系主管确认是否需要重新创建

关联提醒：
  · 该检验单若关联了供应商退货单，需同步 WMS 确认退料状态

来源：QMS 知识库 | Wise:QMS:iqc-submit | 更新：2025-11-15
```

> **飞书富文本消息（post 类型）格式说明**：
> - 文字段和图片段在同一个消息体内自然交替，无需手动拆分拼接
> - 图片通过 `image_key` 内联引用，飞书客户端自动渲染
> - 图片加载失败时，飞书客户端自动显示占位符，不影响文字排版
> - 消息体为 JSON 结构，但比 Interactive Card 简单一个数量级

### 10.2 置信度分档

| 综合分 | 回答方式 |
|--------|---------|
| ≥0.8 | 富文本消息：系统标签 + 步骤（图片内联穿插）+ 来源 + 候选问题 |
| 0.6~0.8 | 同上 + 顶部"⚠️ 仅供参考" |
| <0.6 | 富文本候选列表（Top-3 摘要 + 置信度，让用户追问选择） |
| 无结果 | "暂未收录，已记录您的问题" |
| 降级模式 | 任意档 + 顶部"⚠️ 降级模式" + 置信度×0.7 |
