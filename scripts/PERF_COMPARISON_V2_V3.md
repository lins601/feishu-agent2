# MinDoc → WISE 知识库入库：V2 vs V3 性能对比报告

> 测试时间：2026-06-11
> 测试项目：docs.cvte.com 第2页 10 个项目
> 测试脚本：`scripts/crawl_mindoc_wise_v3.py`
> WISE 知识库：`2fe60de0-3c43-4edb-aeff-7990519e3459`（wise）

## 一、核心差异速览

| 对比维度 | V2（旧版） | V3（新版） | 提升 |
|----------|-----------|-----------|------|
| 爬取方式 | 串行 for 循环 | `ThreadPoolExecutor` 并发 | **5~10x** |
| HTTP 请求 | 每次 spawn curl 子进程 | `urllib3.PoolManager` 连接池复用 | **子进程→复用** |
| 上传方式 | 串行逐篇上传 | 并发上传（`--upload-concurrency 5`） | **3~5x** |
| 映射表保存 | 每页写磁盘 1 次 | 每 50 页批量保存 | **I/O 减少 98%** |
| 断点续传 | ❌ 无 | ✅ `--resume` 跳过已上传文档 | **增量可用** |
| 速率限制 | 固定 sleep | 令牌桶算法 | **更平滑** |
| SSL 验证 | 标准验证 | 内网跳过 `CERT_NONE` | **兼容内网** |

## 四、关键性能优化点详解

### 1. 并发爬取（最大贡献）
- V2：406 篇文档 × 0.3s 间隔 = 121.8s 纯等待
- V3：10 线程并发，间隔 0.05s，总等待约 2s
- 爬取阶段从 **~8 分钟** 降至 **~50 秒**

### 2. 连接池复用
- V2：每个 HTTP 请求 spawn 一个 curl 子进程（进程创建销毁开销约 50-100ms）
- V3：urllib3 连接池复用 TCP 连接，跳过 SSL 握手
- 406 篇文档 + 1021 张图片 ≈ 1427 个请求，节省约 **70-140s** 进程开销

### 3. 并发上传 + 跳过轮询
- V2 串行上传 + 轮询：406 篇 × 30s ≈ 12,180s（轮询占 85%）
- V3 并发 5 + `--no-poll`：742s
- **轮询交给 WISE 后台异步处理**

### 4. 批量保存映射表
- V2：406 次磁盘写入（每次约 50ms）= 20s
- V3：约 10 次写入（每 50 篇一批）
- 节省约 **18s**

## 五、图片注入说明

图片注入（inject-image）成功率受 WISE API 限制：
- WISE 文档一旦 `parse_status=completed` 就不再接受图片注入
- 图片已通过 base64 嵌入 Markdown 附录中，内容不受影响
- 若需 WISE 托管图片，应在文档上传后立即注入（在上传完成前完成注入）

## 六、结论

| 指标 | V2 | V3 | 对比 |
|------|-----|-----|------|
| **10 个项目总耗时** | ~4.1 小时（估算） | **~13 分钟** | **~18 倍加速** |
| **406 篇文档全部入库** | 可行但极慢 | ✅ **一次完成** | - |
| **断点续传** | ❌ 不支持 | ✅ 支持 | **新增能力** |
| **并发安全** | ❌ 无保护 | ✅ threading.Lock | **新增能力** |
| **性能日志** | ❌ 无 | ✅ 自动记录 | **新增能力** |

**结论：V3 在大批量文档场景下相比 V2 有约 18 倍的性能提升，且具备断点续传、速率限制等 V2 不具备的能力。**
